# LogFlux 安全管理功能优化执行方案

> 更新时间：2026-03-06
> 适用范围：`/security` 安全管理模块（前端信息架构、前后端结构优化、发布链路收敛）
> 当前目标：在不回退现有能力的前提下，降低复杂度、提升可维护性与可扩展性

## 1. 背景与问题定义

当前 LogFlux `安全管理` 已经具备较完整的 WAF 控制面能力，覆盖：

- 更新源管理（source）
- Coraza 引擎版本检查（engine）
- 运行策略管理（policy/runtime）
- CRS 调优（crs）
- 规则例外（exclusion）
- 策略绑定（binding）
- 策略观测与误报反馈（observe）
- 版本发布与任务审计（release/job）

从能力覆盖看，系统已经具备“可用的安全控制台”雏形；但从用户体验与代码结构看，当前模块复杂度偏高，主要问题不在“功能缺失”，而在“领域边界不清、入口分散、页面承担职责过多”。

### 1.1 当前主要问题

1. `frontend/src/views/security/index.vue` 体量过大，已成为跨领域的中心编排页。
2. 菜单分组、tab、隐藏子路由并存，导航模型理解成本高。
3. `运行模式 / CRS 调优 / 规则例外 / 策略绑定` 本质都属于同一 `policy` 领域，却以平级 tab 暴露，增加认知负担。
4. `Observe` 同时承载“效果分析”和“误报处置”两套不同工作流，操作过密。
5. `Release / Job / Engine Check / Source` 分散在不同位置，安全变更生命周期被割裂。
6. 前端虽已开始拆 composable，但核心状态编排仍集中于大页面。
7. 前端 API 已出现按域拆分趋势，但 `service/api/caddy.ts` 仍然过于集中。
8. 后端策略发布逻辑功能完整，但编排职责偏重，后续扩展与排障成本高。

### 1.2 优化原则

本次优化遵循以下原则：

- 不回退现有业务能力；
- 尽量不破坏现有 API 合同与数据库结构；
- 先做信息架构和结构重组，再考虑新增能力；
- 优先降低前端单点复杂度；
- 后端以“服务层收敛”优先，而不是立即大规模重写；
- 对旧路由与已有深链尽量保持兼容。

---

## 2. 优化目标

本次优化的核心目标分为 4 类：

### 2.1 用户体验目标

- 让用户按“任务场景”理解安全功能，而不是按底层对象模型理解；
- 降低首次进入安全管理页面的认知门槛；
- 减少跨 tab 来回切换造成的操作割裂；
- 让策略配置、观测、发布、回滚的主路径更清晰。

### 2.2 前端结构目标

- 将超大单页改为“壳层 + 领域容器 + composable”结构；
- 统一导航 schema，减少 route/menu/tab 多份映射配置；
- 将策略相关能力收束为一个工作区；
- 拆分 Observe 的分析与处置流程。

### 2.3 后端结构目标

- 保持现有 `/api/caddy/waf/...` 体系稳定；
- 将策略发布链路中的高复杂度编排下沉到 service；
- 明确策略发布中的校验、配置生成、执行落地、回滚、审计等边界；
- 为后续单测和排障提供更清晰的模块划分。

### 2.4 交付目标

- 输出新的安全管理信息架构；
- 完成前端模块化重构；
- 完成前端 API 按域拆分；
- 完成后端发布链路服务化收敛；
- 保持当前功能闭环可用并通过回归验证。

## 3. 当前能力边界梳理

## 3.1 当前前端领域划分

当前 `/security` 已包含以下业务域：

1. `source`：更新源配置、上传、引擎版本检查
2. `runtime`：运行模式、审计、请求体限制
3. `crs`：PL / anomaly 阈值调优
4. `exclusion`：规则例外配置
5. `binding`：作用域绑定与优先级
6. `observe`：策略统计、趋势、误报反馈、导出
7. `release`：版本发布、激活、回滚
8. `job`：任务审计和清理

## 3.2 当前后端领域划分

当前后端接口与逻辑已具备以下领域：

- `source` 管理
- `engine` 状态检查
- `release` 与 `job`
- `policy` 生命周期
- `policy revision`
- `policy stats`
- `false positive feedback`
- `rule exclusions`
- `policy bindings`

## 3.3 当前主要复杂度来源

- 页面结构复杂度高于功能复杂度；
- 同一领域能力被拆成多个平级入口；
- 路由状态、页面状态、刷新逻辑互相耦合；
- 前后端在“领域边界”上已有雏形，但尚未完全收束；
- 观测与处置、编辑与发布、来源与运维混在同一控制面中。

---

## 4. 目标信息架构

建议将当前安全管理从“多个业务 tab 并列”收束为“4 个一级领域入口”。

## 4.1 一级领域建议

### A. 规则来源

聚焦规则包和引擎供应链入口：

- 更新源配置
- 规则包上传
- Coraza 引擎版本检查

### B. 策略中心

聚焦 WAF 运行策略本身：

- 基础运行模式
- CRS 调优
- 规则例外
- 策略绑定
- 策略预览 / 校验 / 发布 / 回滚
- Revision 历史

### C. 观测与处置

聚焦运行效果和误报治理：

- 效果统计
- 趋势分析
- drilldown 分析
- 误报反馈
- 批量处置
- 导出

### D. 发布运维

聚焦执行与审计：

- Release 管理
- Job 执行记录
- 清理操作
- 回滚记录与运维态查看

## 4.2 设计原则

- 按“用户任务”组织入口，而不是按“数据库对象”组织入口；
- 把同一策略对象的不同配置维度收束到一个工作区；
- 将分析类操作与处置类操作在同一领域内分区，而不是堆叠；
- 保留现有深链兼容能力，但不再强调所有隐藏子路由都是一等导航入口。

---

## 5. 目标前端架构

## 5.1 页面层级结构

建议将安全模块调整为以下结构：

1. `SecurityShell`
   - 负责一级导航
   - 负责路由兼容
   - 负责共享弹窗挂载
   - 负责跨领域刷新事件协调

2. `SecuritySourcePage`
   - 规则来源领域容器

3. `SecurityPolicyPage`
   - 策略中心领域容器

4. `SecurityObservePage`
   - 观测与处置领域容器

5. `SecurityOpsPage`
   - 发布运维领域容器

## 5.2 策略中心重构方向

当前 `runtime/crs/exclusion/binding` 建议收束为“策略工作区”：

- 左侧：策略列表
- 右侧：当前策略工作区
  - 基础设置
  - CRS 调优
  - 规则例外
  - 绑定配置
  - 指令预览
  - 校验/发布
  - Revision 历史

目标收益：

- 用户围绕“一个策略”工作，而不是在四个 tab 间切换；
- 统一策略发布路径；
- 避免 CRS 看起来像独立系统；
- 降低跨模块状态联动复杂度。

## 5.3 Observe 重构方向

建议拆成两个子视图：

### 5.3.1 效果分析

- 概览数据
- 趋势图
- Top Host / Path / Method
- drilldown
- CSV 导出

### 5.3.2 误报处置

- 误报反馈列表
- 状态流转
- 批量处理
- 指派
- 生成 exclusion 草稿

目标收益：

- 区分分析型操作与处置型操作；
- 降低工具栏密度；
- 提高页面聚焦度。

## 5.4 导航配置收敛

当前 route/menu/tab 存在多份 map 配置，建议收敛为一份声明式 schema，统一维护：

- `domain`
- `routeName`
- `title`
- `legacyTabs`
- `defaultChild`
- `visibleInMenu`
- `refreshKey`

目标收益：

- 减少状态同步错误；
- 降低新增/删除入口时的维护成本；
- 让路由逻辑更易理解。

## 5.5 刷新机制收敛

将当前按 tab 分支刷新，改为按领域注册刷新器：

- `source.refresh()`
- `policy.refresh()`
- `observe.refresh()`
- `ops.refresh()`

通过统一的领域刷新映射实现：

- tab 切换刷新
- 发布成功后联动刷新
- 手动刷新入口

目标收益：

- 降低分支判断数量；
- 避免重复 fetch；
- 提高状态一致性。

## 6. 前端实施范围

## 6.1 In Scope

- 安全管理一级信息架构重组
- `index.vue` 壳层化
- 新增领域容器组件
- 策略工作区重构
- Observe 拆分分析/处置视图
- composable 继续按域下沉
- 前端 API 文件按域拆分
- 保持旧路由兼容

## 6.2 Out of Scope

- 不新增高风险安全策略功能
- 不改动现有权限策略
- 不在本阶段替换现有 UI 框架
- 不在本阶段大规模改动后端 API 协议
- 不重做视觉设计系统，仅做必要层级与布局优化

---

## 7. 后端优化方案

## 7.1 保持现有 API 合同稳定

继续沿用当前 `/api/caddy/waf/...` 结构，避免因前端重构带来额外联调风险。

## 7.2 发布链路服务化

围绕策略发布链路拆出 service：

1. `PolicyDirectiveBuilder`
2. `PolicyConfigApplier`
3. `PolicyPublisher`
4. `PolicyRollbackService`
5. `PolicyRevisionService`

### 目标职责划分

- Logic：参数校验、流程编排、响应返回
- Builder：生成 directives
- Applier：构造 candidate config
- Publisher：adapt/load 执行
- Rollback：失败回滚
- Revision：历史快照、revision 持久化
- Notify：通知与审计事件

## 7.3 配置变更边界清晰化

将策略发布拆成 3 个阶段：

### 阶段 1：业务校验
- 策略存在性
- binding 冲突校验
- 参数合法性检查

### 阶段 2：配置生成
- directives 生成
- candidate config 应用
- 配置预览输出

### 阶段 3：执行落地
- adapt
- load
- 失败回滚
- revision 落库
- 历史记录
- 通知发送

目标收益：

- 降低单函数复杂度；
- 明确失败点；
- 更容易补单测与集成测试。

## 7.4 前后端领域边界对齐

后端内部结构逐步向以下领域靠拢：

- source
- policy
- observe
- ops

即便外部 API 路径暂不调整，内部目录和 service 划分也应优先清晰化。

## 8. 分阶段执行计划

## 阶段 P0：方案收敛与基线确认

目标：统一范围、避免重构过程中目标漂移。

交付：

1. 本执行方案评审通过
2. 确认新的一级信息架构
3. 确认旧路由兼容策略
4. 列出需要保留的关键用户路径
5. 梳理回归测试清单

验收：

- 领域划分和页面边界达成一致
- 前后端改造范围明确
- 不新增功能的边界清晰

## 阶段 P1：前端信息架构收束

目标：先解决“用户怎么理解这个模块”。

交付：

1. 一级领域导航落地
2. 旧 tab 到新领域的映射梳理
3. 安全管理首页/壳层重构
4. 旧 deep link 兼容设计

验收：

- 用户可通过 4 个一级入口完成主要任务理解
- 旧路由仍可进入正确领域
- 页面标题、导航高亮、回退行为正确

## 阶段 P2：前端结构拆分

目标：解决 `index.vue` 超大文件问题。

交付：

1. `SecurityShell`
2. `SecuritySourcePage`
3. `SecurityPolicyPage`
4. `SecurityObservePage`
5. `SecurityOpsPage`
6. 统一导航 schema
7. 统一领域刷新器

验收：

- 主页面不再承载具体业务大段模板与逻辑
- 领域容器职责清晰
- 类型检查通过
- 关键操作行为不变

## 阶段 P3：策略中心工作区化

目标：收束策略相关复杂度。

交付：

1. 策略列表 + 工作区布局
2. 运行模式 / CRS / exclusion / binding 收束到同一工作区
3. 发布、预览、校验路径统一
4. revision 与当前策略上下文绑定

验收：

- 用户围绕单一策略完成主要配置流程
- CRS 不再以独立流程形态割裂出现
- 发布链路保持现有能力

## 阶段 P4：Observe 拆分优化

目标：降低观测区混杂度。

交付：

1. 效果分析子视图
2. 误报处置子视图
3. 导出能力归位到分析视图
4. 反馈处置流程独立呈现

验收：

- 分析与处置不再挤在同一操作面板
- URL 状态与页面切换逻辑清晰
- 批量处理与反馈草稿流程保持可用

## 阶段 P5：API 与 composable 真正按域拆分

目标：收束前端模块边界。

交付：

1. `caddy.ts` 逐步瘦身
2. source/policy/observe/ops API 各自承载实现与类型
3. composable 只负责本领域逻辑
4. 页面仅负责展示与事件绑定

验收：

- 新增/修改某一领域时无需频繁触碰中心文件
- 领域边界清晰
- 类型引用路径更直观

## 阶段 P6：后端发布链路服务化

目标：降低核心发布路径复杂度。

交付：

1. publish service 拆分
2. rollback/history/revision service 拆分
3. 发布链路失败点分层
4. 增加关键路径测试

验收：

- 发布逻辑主文件显著收敛
- 失败定位更清晰
- 回滚与 revision 行为保持一致

## 10. 风险与应对

## 风险 1：重构导致已有功能回归

应对：
- 不在第一阶段修改业务语义；
- 保持 API 协议稳定；
- 保留旧路由兼容；
- 建立关键用户路径回归清单。

## 风险 2：前端结构优化中出现状态丢失或重复刷新

应对：
- 优先引入统一导航 schema；
- 再引入统一 refresh registry；
- 通过领域容器逐步迁移，而非一次性推倒重来。

## 风险 3：策略中心工作区化导致交互改动过大

应对：
- 保留原 tab 语义的兼容入口；
- 先做布局合并，再做交互收束；
- 关键动作按钮保持不变。

## 风险 4：后端发布链路拆分影响稳定性

应对：
- 外部接口不变；
- 先提取内部 service，再调整编排层；
- 为失败回滚路径补充测试。

## 风险 5：改造周期长、并行开发互相干扰

应对：
- 分阶段落地；
- 优先前端信息架构和容器拆分；
- 后端发布链路服务化放到后续阶段；
- 每个阶段完成后及时回归。

---

## 11. 验收标准

满足以下条件即视为本轮优化达成：

1. 安全管理一级入口清晰，用户可按任务理解模块结构；
2. `index.vue` 不再承担主要业务逻辑与模板承载职责；
3. 策略相关能力以统一工作区组织，减少跨 tab 认知切换；
4. Observe 的分析与处置流程清晰分离；
5. 前端 API 与 composable 按领域划分更加清晰；
6. 后端发布链路主要复杂度完成服务层收敛；
7. 现有 source / policy / observe / release / job 主流程回归通过；
8. 旧路由与关键深链兼容可用。

## 13. 建议排期

按低风险优先、用户感知优先的顺序，建议排期如下：

1. P0：方案评审与范围冻结
2. P1：信息架构收束
3. P2：前端容器拆分
4. P3：策略中心工作区化
5. P4：Observe 拆分
6. P5：前端 API / composable 按域拆分
7. P6：后端发布链路服务化

---

## 14. 最终结论

当前安全管理的主要问题不是能力不足，而是“多个已完成领域被压缩在单页中，导致认知复杂度与维护复杂度不断放大”。

本次优化建议以“信息架构收束 + 前端壳层化 + 策略工作区化 + Observe 拆分 + 后端服务化”为主线推进，在不破坏现有能力的前提下，将当前安全管理从“功能堆叠页面”升级为“可持续演进的安全控制台”。

该方案优先解决理解成本和结构复杂度问题，为后续继续扩展安全策略、观测分析、误报治理和发布能力提供稳定基础。
